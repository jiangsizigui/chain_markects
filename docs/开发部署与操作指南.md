# 预测市场平台｜开发部署与操作指南

本项目是一个预测市场平台原型，包含：
- **市场创建 + 管理员审核（含 AI 预审）**
- **订单簿式交易撮合（BUY/SELL，YES/NO）**
- **钱包资产/持仓/挂单/成交记录展示**
- **管理员结算（手动/AI）**
- **数据分析（Python 本地引擎 + 前端工作台）**

> 说明：当前撮合模型为 **同一 Outcome（YES 或 NO）内的 BUY/SELL 撮合**：  
> `BUY YES` ↔ `SELL YES`，`BUY NO` ↔ `SELL NO`。  
> “买 YES 与卖 NO 在风险敞口上等价”，但系统不会将 YES/NO 跨方向直接撮合。

---

## 一、环境要求

- **Node.js**：建议 18+（你本机已可运行 22+）
- **Python**：建议 3.10+（用于 `analysis.py`）
- **MetaMask**：用于前端“连接钱包”（地址作为用户标识）

---

## 二、配置环境变量

后端会读取以下环境变量：

- **`ADMIN_SECRET`**：管理员操作密钥（审核市场、结算市场都需要）
- **`GEMINI_API_KEY`**：用于管理员 AI 审核、AI 结算（可选，不填则 AI 功能不可用）

### Windows PowerShell 示例

```powershell
setx ADMIN_SECRET "your-admin-secret"
setx GEMINI_API_KEY "your-gemini-api-key"
```

设置后请 **关闭并重新打开终端** 再运行项目。

---

## 三、开发启动

在项目根目录执行：

```bash
npm install
npm run dev
```

默认后端监听端口为 **3000**（`server.ts` 内，可通过环境变量覆盖）。  
如果遇到端口占用或权限问题（例如 Windows 上 `EACCES 127.0.0.1:3000`）：

1) 检查占用：

```powershell
netstat -ano | findstr ":3000"
```

2) 推荐用环境变量改端口启动（示例 3100）：

```powershell
cmd /c "set PORT=3100&& npm run dev"
```

启动成功后在浏览器打开：
- `http://localhost:3100`

## 四、核心操作流程（从创建到结算）

### 1）创建市场（用户侧）

路径：**市场大厅 → 创建市场**

填写：
- 市场问题标题
- 事件描述
- 分类
- 截止时间
- 结算数据来源

提交后市场状态为：**`PENDING（待审核）`**。

---

### 2）管理员审核市场（人工 / AI）

路径：**管理员 → 待审核市场**

操作：
- **初始化管理员白名单（仅首次）**：
  - 在“管理员初始化”卡片中输入 `ADMIN_SECRET`
  - 点击“将当前地址加入管理员白名单”
  - 成功后再点击右上角“钱包签名登录管理员”
- **钱包签名登录管理员**：
  - 点击右上角“钱包签名登录管理员”
  - 系统会让钱包对固定消息进行签名，后端验证后颁发 12 小时有效的管理员令牌
- 点击 **AI 审核**：后端调用大模型输出 JSON 建议（APPROVE/REJECT）
- 点击 **通过 / 拒绝**：执行人工审核
- 如果 AI 审核结果可解析为结构化 JSON，会显示“一键采用建议并执行审核”

通过后市场状态变为：**`OPEN（交易中）`**。

---

### 3）多账号真实交易撮合（A/B 账户示例）

准备：
- 使用 MetaMask **切换两个不同地址**（或使用两个浏览器 Profile）

撮合规则（重要）：
- 要成交必须是 **同一 Outcome** 的 **BUY/SELL 对向**：
  - A：`BUY YES` 需要 B：`SELL YES`（且价格满足撮合条件）
  - A：`BUY NO` 需要 B：`SELL NO`

推荐测试步骤：
1. 账号 A 进入某市场，挂 `BUY YES`（买入 YES）
2. 切换到账号 B，挂 `SELL YES`（卖出 YES），价格 ≤ A 的买价
3. 若价格可成交，撮合立即发生：
   - 生成成交记录（Trade）
   - A/B 钱包与持仓更新
   - 市场 YES/NO 价格随成交更新

查看结果：
- **钱包管理**：
  - 当前余额 / 冻结资金
  - 当前持仓（YES/NO 份额与冻结）
  - 个人“我的交易记录”表格
  - “历史累计净现金流”曲线
- **市场详情**：
  - “我的挂单”（OPEN 订单）
  - “我的持仓”
  - 订单簿（YES）

---

### 4）用户卖出持仓并成交

在市场详情页：
- 使用 **卖出 YES/卖出 NO** 挂单（系统会检查持仓是否足够）
- 另一个账号挂对向买单，触发撮合

成交后：
- 卖方钱包余额上升
- 持仓减少
- Trade 记录增加（Wallet 的交易记录表格可查）

---

### 5）市场到期与结算

市场在超过 `endTime` 后：
- 后端会自动把 **`OPEN` → `CLOSED（待结算）`**
- 前端交易按钮会禁用并提示“已截止待结算”

管理员结算路径：**管理员 → 待结算市场**

结算方式：
- **手动结算 YES/NO**
- **AI 结算（联网检索）**
  - **AI 辅助人工**：AI 联网检索证据后给出 YES/NO 建议 + 原因 + 来源链接，管理员复核后“一键采用并结算”
  - **纯 AI**：AI 联网检索后直接结算，并将“公示依据”写入市场详情供社区复核

结算后：
- 市场进入 `RESOLVED`
- 系统按最终结果向获胜方向持仓发放收益（简化模型：1 PMT/份）
- 清空该市场持仓与挂单（并退还未成交部分的冻结资金/份额）

## 五、数据分析模块（Python 引擎）

路径：**数据分析**

- 上传 `.json/.csv/.xlsx` 数据
- 选择模型（8 类模型）
- 点击运行：前端调用 `POST /api/analyze`，后端会执行 `analysis.py` 并返回结果

> 注意：需要本机可用 `python` 命令（加入 PATH）。

### 1）8 个模型列表（当前实现）

- **ARIMA 时间序列趋势预测**（`model=arima`）
  - 输入：`data=[...]`（价格/概率序列），可选 `horizon`
  - 输出：未来序列预测、趋势（上升/下降/持平）
- **线性回归**（`model=linear_regression`）
  - 输入：`data_x=[...]`、`data_y=[...]`
  - 输出：斜率/截距/R²/方程
- **贝叶斯更新**（`model=bayesian`）
  - 输入：`prior`、`likelihood`，可选 `evidence`
  - 输出：`posterior` 与变化量
- **市场情绪指标**（`model=sentiment`）
  - 输入：`buy_volume`、`sell_volume`
  - 输出：`sentiment_score` 与偏多/偏空/中性
- **K-Means 聚类**（`model=kmeans`）
  - 输入：`k`、`points=[[...],[...]]`
  - 输出：簇中心、标签、各簇样本数
- **GARCH 波动率**（`model=garch`）
  - 输入：`data=[...]`（价格/概率序列）
  - 输出：`volatility_score` 与风险等级
- **定价偏差（Model vs Market）**（`model=mispricing`）
  - 输入：`model_probability`、`market_probability`（0-1）
  - 输出：`edge` 与机会提示
- **相关性分析（皮尔逊）**（`model=correlation`）
  - 输入：`series_a=[...]`、`series_b=[...]`
  - 输出：`correlation` 与强度标签

### 2）上传文件格式示例

- **单序列（ARIMA/GARCH）**：CSV 单列或 JSON 数组
  - JSON：`[0.63, 0.64, 0.62, 0.66, 0.67]`
- **双序列（线性回归/相关性）**：CSV 两列或 JSON `{x:[], y:[]}`
  - JSON：`{"x":[10,20,30],"y":[0.55,0.60,0.66]}`
- **多维点（KMeans）**：CSV 三列及以上或 JSON `{points:[[...],[...]]}`
  - JSON：`{"points":[[5,100],[30,200],[6,120]]}`

## 六、常见问题

### 1）为什么“买 YES”和“买 NO”不会互相成交？

系统采用订单簿撮合：同一 Outcome 内 BUY/SELL 对向撮合。  
“买 YES 与卖 NO 等价”是风险敞口等价，不代表订单簿会跨 Outcome 直接撮合。

### 2）市场已截止，但还能看到价格/订单吗？

可以。市场 `CLOSED` 后不允许新下单，但仍可查看历史数据、挂单状态（会在结算时统一取消退还）。

## 七、生产部署建议（原型）

当前后端为内存数据库（重启会丢数据）。若要更像“可部署系统”，建议：
- 将 markets/orders/trades/wallets/positions 持久化到数据库（如 SQLite/Postgres）
- 增加管理员认证（不仅是 `ADMIN_SECRET`）
- 使用 Nginx/PM2 或容器化部署
